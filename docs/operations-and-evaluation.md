**运维支撑与评估管线：外围设施详解**

智能体团队的协作内核之外，一组辅助性模块为整套流程的落地运转提供必要的外围支撑。它们的关注面从物理层面的依赖部署延伸至实验层面的效能审计，贯穿了从环境准备、作业启动、进度感知到成果检验的完整操作链路。这些设施的直接消费者不再是框架中的智能体角色，而是驾驭系统开展评测实验的研究人员。

```
codes/
├── setup_evaluation_env.py          # 地理空间依赖链的自动化构建
├── run_iterative_repair.py          # 批量修复流程的命令行触发点
│
└── evaluator/
    ├── __init__.py
    ├── workspace_manager.py         # 任务元数据的解析与多条件筛选
    ├── dialogue_manager.py          # 任务终态的轻量级状态注册
    ├── task_evaluator.py            # 单任务执行历程的LLM驱动评价
    └── summary_reporter.py          # 跨任务维度的汇总报告生成
```

**5. `task_evaluator.py`——调试效能的模型辅助审视**

单任务评估器在任务抵达终态后自动触发，从四个维度汇集分析素材：初始脚本与最终版本的代码比对、数据探查员的结构审查报告与工程师的设计文档、团队成员间的请求-响应交互实例、以及诊断专员经上下文压缩后的完整行动序列。

交互实例的提取遵循配对原则——在接收方的消息序列中定位特定类型的入站请求（`data_request`、`patch_submission`、`inject_request`），向下搜索至与之呼应的`task_report`回复，将这段区间内的行动片段切取出来，转化为带有动作链标注的可读文本嵌入评估提示词。诊断专员的完整序列则以JSON原貌直接编入，保留工具调用与推理摘要的全部细节。

评估提示词要求模型从交付质量、数据探查效能、工程规划与落地的吻合度、协作沟通的精准性、诊断策略的合理性五个切面展开分析，最终产出一份中文撰写的Markdown评估报告，以元数据头部加正文分析的形态存储于任务输出目录。
