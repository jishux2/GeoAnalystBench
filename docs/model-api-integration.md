**模型推理接口：技术参考与服务端对接**

**协议基座**

项目中的全部模型交互——无论服务于智能体团队的工具调用循环还是评估管线的单轮判定——均建立在OpenAI Chat Completions协议之上。这套协议规定了请求与响应的规格契约，任何声称兼容该规范的服务端都可在不改动客户端代码的前提下互换接入。

请求体的核心载荷是一条按时序编排的消息序列，每个元素由角色标识与内容主体构成。角色涵盖四种类型：`system`承载全局性的行为框架与身份界定，`user`代表外部输入的指令或素材，`assistant`记录模型先前的应答轨迹，`tool`回填工具执行后的产出。序列中各角色按交互的实际发生顺序交替排列，还原多轮对话的完整脉络。

内容主体的表达形式因消费场景而分化。最基本的形态是纯文本字符串，适用于绝大多数的指令传达与文本生成场景。当需要在同一条消息中混合文本与非文本媒介时，内容主体可切换为一个类型化的内容块数组，每个块通过`type`字段标明自身的介质属性：

```json
{
  "role": "user",
  "content": [
    {"type": "text", "text": "描述这张图中的空间分布特征。"},
    {"type": "image_url", "image_url": {"url": "data:image/png;base64,..."}}
  ]
}
```

文本块与图像块可在数组中自由穿插排列，模型按序逐块处理，对文本执行语义理解、对图像展开视觉感知，两种模态的信息在推理过程中交汇融合。图像块中的URL既接收指向外部资源的网络地址，也容纳以Base64编码内联的Data URI——后者在无法假设网络可达性的本地评测环境中尤为实用。需要留意的是，并非所有兼容此协议的服务端都完整支持内容块数组形式；纯文本字符串作为内容主体的表达则具备普遍的适配性。

请求体中的其余参数调控生成行为的各个侧面。`model`指定目标模型标识符，其取值决定默认的输出行为与容量配额。`temperature`调节采样过程的随机程度，数值偏低时输出趋于确定性，偏高则引入更多变异。`max_tokens`划定单次响应的长度天花板。`stop`定义终止序列，模型在输出匹配该序列的文本时立即截断生成。`stream`以布尔值控制传输模式——设为`false`时服务端在生成全部完成后一次性返回；设为`true`时响应以服务端推送事件的流式分片逐步抵达，每个分片携带一个增量`delta`对象而非完整的`message`结构，客户端需自行拼接以还原最终产出。

响应体中的核心内容嵌套于`choices[0].message`路径下（流式模式中对应为`choices[0].delta`）。`content`字段承载生成文本，`tool_calls`数组在模型决定援引外部函数时浮现，两者可在同一响应中共存。

**DeepSeek服务端**

DeepSeek平台维护两套服务入口：

| 入口类别 | 地址 | 适用范围 |
|---------|------|---------|
| 标准入口 | `https://api.deepseek.com` | 对话补全、工具调用、思考模式 |
| Beta入口 | `https://api.deepseek.com/beta` | 试验性功能（含前缀续写、严格模式等） |

模型层面提供两个标识符。`deepseek-chat`对应常规对话模式，不主动展开推理链，默认输出上限为4K token，可经`max_tokens`调至8K。`deepseek-reasoner`为同一底座的思考模式变体，在生成最终回答前先输出一段显式的推理过程，默认输出容量扩展至32K，上限可达64K。值得注意的是，思考模式并非后者的专属能力——通过在请求体中配置`thinking`参数为`{"type": "enabled"}`，`deepseek-chat`同样能够激活推理链输出，允许开发者在单一模型标识符下灵活切换两种工作状态。

上下文窗口容量为128K token，涵盖输入序列与输出生成的合计消耗。这一额度构成了智能体上下文管理策略——推理链压缩的临界阈位、工具结果的体积预算——的底层物理约束。

*工具调用机制*

工具调用赋予模型在对话过程中主动触发外部函数的能力。`tools`参数接收一组函数声明，每个条目描述一项模型可支配的操作资源，涵盖入口名称、职能阐释与参数的JSON Schema约束。模型依据对话上下文与函数描述的关联程度自行判断是否发起调用以及选取哪个函数，单次请求至多声明128个函数。

`tool_choice`参数从调用方视角施加行为管控：

| 取值 | 效果 |
|------|------|
| `none` | 完全禁止工具介入 |
| `auto` | 授权模型自主抉择 |
| `required` | 强制模型必须激活至少一项工具 |
| `{"type": "function", "function": {"name": "..."}}` | 锁定特定函数 |

工具调用场景下的对话序列遵循固定的递进规程：模型决定调用函数时，响应中的`tool_calls`数组呈报函数名、序列化的参数包体以及系统分配的调用标识；调用方执行实际逻辑后将结果以`tool`角色消息回填，`tool_call_id`与先前的标识配对；扩展后的序列再次递交模型以驱动下一步决策。如此往复直至模型给出最终的文本回答。

思考模式可与工具调用并行运作。启用后响应中额外呈现`reasoning_content`字段记录中间思考轨迹。几项配套约束值得留意：`temperature`等采样参数在思考模式下被静默忽略；`max_tokens`的预算涵盖推理链与最终输出的合计消耗；历史消息中累积的`reasoning_content`会持续占用上下文窗口，官方推荐在构造后续请求前将已处理轮次的推理内容予以剥离。最后一点在智能体的长程交互场景中尤为关键——每轮工具调用都会扩展消息序列，若不及时清理推理文本，上下文膨胀的速度将远超有效信息的积累。

*前缀续写*

Beta入口开放了一项输出约束能力：在消息序列末尾附加一条标记`prefix: true`的assistant消息，其内容被视为模型回复的既定开头，后续生成从该锚点接续展开。搭配终止序列可将产出精确框定在预设的起止边界之内。当前系统的代码编写由脚本工程师在工具调用循环中自主完成，未采用此机制。

*严格模式*

Beta入口还引入了Strict Mode作为工具调用的格式保障层。在函数声明中将`strict`属性设为`true`后，API强制校验模型输出的参数结构与预定义Schema的零偏差对齐。启用时须遵守一项附加约束：所有`object`类型的属性必须声明为`required`，且须将`additionalProperties`设为`false`。

*计费与并发*

按token计量的阶梯模式：输入侧依据缓存命中情况区分定价，命中时0.2元/百万token，未命中则为2元/百万token；输出侧统一按3元/百万token结算。并发层面未对单用户施加固定的请求频率约束，仅在服务端承压时通过503状态码示意调用方适度退让。

*错误码体系*

API调用过程中遭遇的异常通过HTTP状态码传递分类信号。客户端错误源于请求自身的配置缺陷，重试无法改善状况：

| 状态码 | 语义 | 典型成因 |
|--------|------|---------|
| 400 | 请求体格式畸形 | JSON损坏或必填字段缺失 |
| 401 | 身份凭证无效 | API密钥错误或过期 |
| 402 | 账户额度耗尽 | 余额不足 |
| 422 | 参数约束违规 | 字段取值超出许可范围 |

服务端错误反映平台侧的临时性状况，经适当等待后通常可自行恢复：

| 状态码 | 语义 | 应对姿态 |
|--------|------|---------|
| 429 | 请求频率过密 | 突发流量唤起保护机制 |
| 500 | 内部处理故障 | 服务端运行时异常 |
| 503 | 负载容量饱和 | 系统资源暂时紧张 |

针对后一类状况，指数退避策略在实践中表现稳健——连续失败之间的等待间隔逐次翻倍，通过递增的冷却周期缓解对服务端的集中冲击。网络中断与请求超时同样纳入退避重试的容错范畴。

**评估端点**

评估管线接入一个本地部署的反向代理服务，端点地址为`http://127.0.0.1:8317`，在协议层面完全遵循前述的Chat Completions规范，与DeepSeek API的调用逻辑可互换复用。两者的分野体现在三个维度上。

在模型标识的维度上，评估端点背后承载的`gpt-5.4`具备原生的多模态感知能力，能够在接收图像内容块时直接实施视觉理解，而无需将图像预先转译为文本描述。这一特性使得评估环节可以将智能体产出的地图、图表等视觉产物以Data URI的形式嵌入消息中，令评估模型在文本分析与图像审视之间无缝切换。

传输层面，该端点仅支持流式响应，客户端须以`stream: true`发起请求并逐片拼接增量内容。每个到达的数据分片遵循服务端推送事件的编码约定，以`data: `为前缀托载JSON分片，流的终结以`data: [DONE]`标记收束。

身份验证沿用与DeepSeek相同的Bearer Token机制，密钥为该反代服务的专属凭证，与DeepSeek API密钥相互独立。
