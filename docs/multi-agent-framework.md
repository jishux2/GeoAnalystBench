**通信原语与消息路由**

智能体之间的一切协同行为最终都归结为消息的发送与接收。框架在这一层面所确立的基础设施，既为团队沟通中涌现的多元诉求提供了统一的结构化表达，也为这些表达在异步环境中的可靠送达铺设了物理通路。

**1. `message.py`——协作意图的类型化表达**

每条在成员间流转的消息都是一个自描述的数据实体，承载六项属性：语义类别、发送方与接收方的身份标识、自由文本形式的正文、可选的机器可读载荷，以及用于时序排列的时间戳与唯一标识。

语义类别通过枚举定义，覆盖了团队协作中出现的全部意图光谱：

| 类别 | 流向 | 用途 |
|------|------|------|
| `task_assignment` | 主控节点 → 成员 | 分派工作指令与任务上下文 |
| `task_report` | 成员 → 成员/主控节点 | 交付阶段性成果或回复请求 |
| `data_request` | 任意成员 → 探查员 | 索取数据层面的针对性审查 |
| `patch_submission` | 诊断专员 → 工程师 | 递交代码修改指令 |
| `inject_request` | 诊断专员 → 工程师 | 请求植入诊断语句并执行 |
| `status_inquiry` | 主控节点 → 诊断专员 | 问询当前调查进展 |
| `status_reply` | 诊断专员 → 主控节点 | 反馈工作状态概要 |
| `task_complete` | 诊断专员 → 主控节点 | 宣告任务目标达成或提交受迫结论 |
| `terminate` | 主控节点 → 成员 | 指示成员终止运行 |

消息实体同时提供两种序列化出口：`to_context_text`将自身渲染为适合注入智能体对话序列的文本片段，载荷部分以格式化的JSON代码块呈现以便模型识别其中的结构化语义；`to_log_dict`则输出面向磁盘归档的精简字典，仅保留载荷的键名清单而非完整内容，控制日志体积。

**2. `channel.py`——异步队列驱动的投递网络**

每位成员在注册加入团队时获得一条专属的收件通道，其内核是一个无界的`asyncio.Queue`实例。通道对外暴露三种访问姿态：非阻塞的即时投递、可设超时的阻塞式等待、以及一次性排空当前积压的批量提取。第一种服务于发送方的零延迟投递需求，第二种为主控节点的周期性苏醒提供带超时的挂起语义，第三种供智能体在每轮工具调用间隙批量检查收件箱。

全局的通道注册表维护从成员名称到通道实例的映射关系。消息投递时，注册表依据消息中的接收方字段检索目标通道并完成路由——发送方无需持有对方通道的直接引用，只需知晓其名称标识即可触达。这一间接寻址的设计将成员间的耦合度压缩至最低限度：任何一方的加入或退出都不影响其余成员的通信能力，只要注册表中的映射保持更新。

**4. `base_agent.py`——协程骨架与基础能力集**

所有子智能体共享的运行时框架凝聚于此。核心是一个围绕`while not self._terminated`展开的异步循环，每个迭代周期依次完成收件箱排空、上下文容量评估、API调用、工具执行、快照持久化五个步骤。模型若未给出工具调用指令而是产出纯文本回复，循环直接进入下一周期；若模型选择了`go_idle`工具，协程挂起于收件通道的阻塞式等待，直至新消息到达后恢复运转。收到`terminate`类型的消息时终止标记被置位，循环在当前周期结束后自然退出。

骨架预装了六项基础工具，构成所有角色共享的最小能力底座：

- `load_skill` / `unload_skill`——技能的加载与卸载，前者触发知识注入和工具扩充，后者执行反向清理
- `send_message`——向指定成员投递消息，载荷接纳任意合法的JSON结构
- `go_idle`——显式声明进入空闲态，挂起至下一条消息抵达
- `read_file` / `close_file`——文件内容的按需获取与上下文释放，前者支持可选的行号标注以辅助定位

角色特化通过三个扩展点注入：`build_system_prompt`提供身份提示词，`configure`执行启动前的额外配置，`on_activated`在循环开始前触发首要动作。子类覆写这些方法即可在不触碰循环骨架的前提下定制各自的行为特征。

**Agent Skills技能规范**

框架采用Anthropic提出的Agent Skills标准来组织智能体的领域知识供给——一种将操作指南与配套资源封装为可按需装载的标准化指令包的规范。每项技能在文件系统中表现为一个自含的目录，其下的`SKILL.md`承载核心方法论，`scripts/`目录存放预置的可执行诊断程序。

`SKILL.md`文件由两个区域构成。开头的YAML前置区以`name`和`description`两个字段提供机器可读的摘要，供技能注册表在启动扫描时快速建立能力索引；其后的Markdown正文铺陈完整的操作方法论——职责界定、工作流编排、工具使用要领、团队协作契约等实质性内容均在此承载。注册表索引的是前者，注入上下文的是后者，二者各司其职。

智能体启动时仅凭摘要信息即可纵览全部可用技能，具体正文留待模型通过`load_skill`工具主动激活后才进入对话序列。这种延迟加载的策略在信息充裕与上下文经济之间取得平衡——成员不会在尚未确定工作方向时就被大段的领域文档占据宝贵的窗口空间。
