**通信原语与消息路由**

智能体之间的一切协同行为最终都归结为消息的发送与接收。框架在这一层面所确立的基础设施，既为团队沟通中涌现的多元诉求提供了统一的类型化表达，也为这些表达在异步环境中的可靠送达开辟了物理通路。

**1. `message.py`——协作意图的结构化编码**

每条在成员间流转的消息都是一个自描述的数据实体，以dataclass形态承载六项属性：语义类别、发送方与接收方的身份标识、自由文本形式的正文、可选的机器可读载荷，以及用于时序排列的时间戳与唯一标识。

语义类别通过枚举穷举，覆盖协作过程中出现的全部意图光谱：

| 类别 | 流向 | 用途 |
|------|------|------|
| `task_assignment` | 主控节点 → 成员 | 分派工作指令与任务上下文 |
| `task_report` | 成员 → 成员/主控节点 | 交付阶段性成果或回复请求 |
| `data_request` | 任意成员 → 探查员 | 索取数据属性的针对性审查 |
| `status_inquiry` | 主控节点 → 诊断专员 | 问询当前调查进展 |
| `status_reply` | 诊断专员 → 主控节点 | 反馈工作状态概要 |
| `task_complete` | 诊断专员 → 主控节点 | 宣告任务目标达成或提交受迫结论 |
| `terminate` | 主控节点 → 成员 | 指示成员终止运行 |

载荷字段采用开放类型声明，接纳任意合法的JSON构造——对象、数组及其嵌套组合均可入驻，以适应不同类别对附加数据的差异化排布需求。

消息实体同时提供两种序列化出口。`to_context_text`将自身渲染为适合注入对话序列的文本片段，载荷部分以格式化的JSON代码块呈现以便模型识别其中的语义结构；`to_log_dict`则输出面向磁盘归档的精简字典，仅保留载荷的键名清单而非完整内容，控制日志体积。

**2. `channel.py`——协程信箱编织的投递网络**

每位成员在注册加入编队时获得一条专属的收件通道，其内核是一个无界的`asyncio.Queue`实例。通道对外暴露三种访问姿态：非阻塞的即时投递、可设超时的阻塞式等待、以及一次性排空当前积压的批量提取。第一种服务于发送方的零延迟投递需求，第二种为主控节点的周期性苏醒提供带超时的挂起语义，第三种供智能体在每轮工具调用间隙批量检查收件箱。

全局的通道注册表维护从成员名称到通道实例的映射关系。消息投递时，注册表依据消息中的接收方字段检索目标通道并完成路由——发送方无需持有对方通道的直接引用，只需知晓其名称标识即可触达。这一间接寻址的设计将成员间的耦合度压缩至最低限度：任何一方的加入或退出都不影响其余成员的通信能力，只要注册表中的映射保持更新。

**3. `tool_base.py`——能力声明与调度枢纽**

框架中的每项可用操作都以`ToolSpec`数据类的形态存在，将面向API的接口契约（名称、功能描述、参数的JSON Schema）与实际的执行逻辑（一个异步可调用对象）绑定为不可分割的整体。`ToolDispatcher`管理当前会话中已注册的全部工具实例，对外提供按名称检索与执行的统一入口，对内支持运行时的动态注册——技能加载时工厂函数产出的工具实例经由此通道汇入可用集合。

调度器同时肩负工具输出的体积管控。每项工具可被赋予独立的容量配额，未经显式指定者采纳全局默认水位；自带流量节制的工具——如文件读取——可获标记豁免，不受此道关卡约束。超限产出的落盘归档随智能体的输出目录组织，以递增编号串联时序，便于事后按需回检。

在每轮API请求中，调度器将已注册工具的Schema声明导出为符合DeepSeek协议的数组格式，随对话序列一同递交；模型返回的工具调用指令经JSON反序列化后，由调度器按名称匹配到对应的处理方法并转发执行。这一闭环将格式约束的职责完全下沉至API协议层，上层逻辑无需介入任何文本解析工作。

**4. `base_agent.py`——协程骨架与基础能力集**

所有子智能体共享的运行时框架凝聚于此。核心是一个围绕`while not self._terminated`展开的异步循环，每个迭代周期依次完成收件箱排空、上下文容量评估、API调用、工具执行、持久化五个步骤。模型若未给出工具调用指令而是产出纯文本回复，循环直接进入下一周期；若模型选择了`go_idle`工具，协程挂起于收件通道的阻塞式等待，直至新消息到达后恢复运转。收到`terminate`类型的消息时终止标记被置位，循环在当前周期结束后自然退出。协程在运行过程中若遭遇未预期的异常，会在终止前向主控节点发送携带故障摘要的紧急通知，确保编队层面能够及时感知成员的非正常退出。工具调用指令的参数反序列化亦设有容错兜底，格式畸变时以错误描述回填对话序列而非令协程崩溃，给予模型自行修正的机会。

骨架预装了六项基础工具，构成所有角色共享的最小能力底盘：

- `load_skill`——技能的按需激活，触发知识注入和工具扩充
- `send_message`——向指定成员投递消息，载荷接纳任意合法的JSON结构
- `go_idle`——显式声明进入空闲态，挂起至下一条消息抵达
- `read_file`——工作空间内文本文件的受控读取，兼顾精确定位与容量约束
- `grep`——ripgrep赋能的全局搜寻，兼供内容扫查与目录探测
- `compact_context`——主动将推理链送入归纳与精简流程，腾出上下文空间

读取工具对命中数据集仓库管辖范围的路径部署准入屏障，以指向性提示取代文件内容返回，将数据源的审查职责收归于探查员的专业工具链。

基础的序列操作涵盖用户消息追加、模型响应拼接、工具结果回填，以及技能正文和团队消息的注入——后两者以带有明确边界标记的用户消息形态嵌入，技能文档额外携带标签字段以支持后续的精确定位。

**Agent Skills技能规范**

框架采用Anthropic提出的Agent Skills标准来组织智能体的领域知识供给——一种将操作指南与配套资源封装为可按需装载的标准化指令包的规范。每项技能在文件系统中表现为一个自含的目录，其下的`SKILL.md`托管核心方法论，`scripts/`目录存放预置的可执行诊断程序。

`SKILL.md`文件由两个区域构成。开头的YAML前置区以`name`和`description`两个字段提供机器可读的摘要，供技能注册表在启动扫描时快速建立能力清单；其后的Markdown正文铺陈完整的操作方法论——职责界定、工作流梳理、工具使用要领、团队协作契约等实质性内容均在此呈现。机器侧的扫描程序从中摘取结构化元数据以填充能力目录，而模型在加载技能时所迎接的则是一份贯穿全局的实践蓝图——两个区域各自服务于不同阶段的信息诉求。

智能体启动时仅凭摘要信息即可纵览全部可用技能，具体正文留待模型通过`load_skill`工具主动拉取后才汇入对话脉络。这种延迟揭示的策略在信息充裕与上下文经济之间取得平衡——成员不会在尚未明确工作方向时就被大段的领域文档挤占珍贵的认知带宽。

**13. `coordinator.py`——团队生命周期的管控中枢**

主控节点的内部运作以一个单层状态机为骨架。监控循环在每个三十秒的轮询窗口中依次执行三项检查：从自身收件通道提取新到达的消息并按`MessageType`枚举值分派至对应的处理分支，遍历各子任务的`asyncio.Task`句柄以侦测是否有协程已携带异常终结，最后将累计耗时与全局预算比对以判定是否触及超时边界。三项检查的顺序确保了消息响应优先于故障侦测，而故障侦测又优先于超时裁定——一个恰好在超时临界点完成修复的诊断专员，其终态消息会在同一轮循环中被优先消费，避免被随后的超时逻辑误判为未完成。

消息处理的分支中，两类事件承担着关键的状态转移职能。工程师发出的首条`task_report`触发对`current_script.py`的快照留存，这份副本在任务收官时作为diff运算的基线端参与变更记录的生成。诊断专员递交的`task_complete`携带业务评估与根因分析载荷，其抵达直接驱动监控循环的正常退出。

超时终止遵循一套渐进式的收束策略。首先向诊断专员投送终止指令，随后开启一个短暂的宽限窗口等待其受迫提交——指令措辞中明确告知：若当前恰有未经验证的修补在手，应先执行一轮再作答复。宽限窗口关闭后，节点向全部存活成员广播终止消息并以`asyncio.wait`设定上限等待协程退出，逾期未响应者由`Task.cancel`强制回收。残存的PDB调试会话在最后阶段经由三级退出序列（常规quit→二次quit应对嵌套层级→进程终止信号兜底）完成清理。
