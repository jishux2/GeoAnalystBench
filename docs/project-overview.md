**GeoAnalystBench多智能体修复系统：项目概览与架构纵览**

**项目背景与研究动机**

GeoAnalystBench是一套面向大语言模型空间分析能力的基准评测体系，汇集了50个经GIS领域专家审定的地理处理任务。这些任务横跨城市热岛效应建模、野生动物迁徙追踪、地下水脆弱性评估、交通事故聚集识别等多元应用场景，旨在考察模型能否依据自然语言描述产出可在真实数据集上运行的Python程序。

任务设计遵循ESRI空间分析分类框架，沿六个方法论维度铺展：理解位置（U）、测量形态分布（M）、确定位置关联（DR）、寻找最优方案（F）、检测量化模式（DP）、空间插值预测（S）。每项任务可同时归属至多三个类别，以映射地理分析工作的复合特质。技术栈层面则显式区分了开源工具链（geopandas、rasterio等）与闭源ArcPy接口两条路线，为不同技术生态下的模型表现提供对照基准。

本项目在原始评测体系之上，进一步构建了一套基于多智能体协作的自动化修复系统。其核心关切并非"模型能否一次性生成正确代码"这一静态命题，而是延伸至一个更贴近真实软件工程实践的问题域：当生成的脚本存在缺陷时，能否通过多个专职角色的分工协作——数据资源的结构性探查、脚本的迭代编写与执行、运行时故障的系统性诊断——在持续的信息交换与证据积累中逐步收敛至可行解。这一视角将评测重心从孤立的代码生成质量，转向了团队化智能体在开放性调试任务中的协调能力与问题解决效率。

**基准测试集的构成**

全部评测素材集中于`dataset/GeoAnalystBench.csv`文件，每条记录对应一个独立的地理处理任务。以下字段构成驱动后续各环节运转的核心信息：

| 字段 | 含义 |
|------|------|
| id | 任务的数值编号 |
| Open Source | 技术栈标识，`T`代表开源工具，反之依赖闭源环境 |
| Task | 对任务目标的简要概述 |
| Instruction | 完成任务所需遵循的详细操作指引 |
| Domain Knowledge | 辅助理解任务的领域专业知识 |
| Dataset Description | 所涉数据文件的名称、格式及关键字段说明 |
| Task Length | 专家设计的标准工作流包含的步骤数量 |
| CodeString | 专家编写的参考Python实现 |

方法论归属通过Task Categories1至Task Categories3三个附加列标注，采用前述六维分类体系的完整英文描述进行编码。单个任务至多横跨三个类别，未占满的列位留空处理。

在系统运行中，任务信息提取器从Task、Instruction、Domain Knowledge、Dataset Description四个字段中萃取文本素材，组装为递交智能体团队的完整任务描述；技术栈标识则左右着脚本执行时所启用的Python解释器环境——开源任务路由至专属虚拟环境，闭源任务指向ArcGIS Pro内置的运行时。而CodeString中的专家实现则在环境搭建阶段发挥辅助效用：通过扫描其中的导入语句，自动汇总所需第三方库以构建完备的依赖清单。

**评测范式**

系统围绕"团队协作修复"的核心理念运作。每个地理处理任务被交付给一个由四名成员组成的智能体小组：一位数据探查员负责摸清数据资源的结构底貌，一位脚本工程师承担从方案构思到代码落地的全链路实现，一位诊断专员专司运行时缺陷的定位与修复方案拟定，以及一个主控节点统筹协调整体节奏。

任务启动后，探查员与工程师并行进入各自的预备阶段。前者对数据集中的每份文件执行格式化的结构审查——字段清单、几何特征、坐标参考体系、值域分布——将发现汇编为一份磁盘报告后移交工程师参考。后者在等待数据细节的同时先行推进抽象层面的技术规划，待报告到达后整合具体的字段引用与路径配置，完成脚本编写并提交首次执行。诊断专员在此期间保持待命，直到工程师的执行产物——脚本文件连同运行时输出——抵达其收件箱。

诊断环节构成了修复过程的核心迭代。专员通过阅读结构化的异常追踪记录和函数调用日志切入调查，借助交互式调试器在崩溃现场或特定代码区域探查运行时状态，逐步缩小故障范围。当形成修复方案时，它将代码变更指令发送给工程师执行并等待回报；若需在不修改源文件的前提下观测中间变量，则请求工程师生成一份嵌入诊断语句的临时副本加以运行。数据层面的疑点则路由至探查员进行针对性复核。这一"假设—验证—修正"的循环持续推进，直至脚本通过全部运行时校验且业务输出符合任务预期。

主控节点在大部分时间保持静默，周期性地向诊断专员问询进展以维持对任务走向的感知。当诊断专员确认任务目标已达成时，它向主控节点递交一份涵盖业务评估与根因分析的终结报告，后者随即启动团队的有序关闭。若全局时间预算先行耗尽，主控节点主动介入，要求诊断专员在限期内提交附带置信度标记的当前最佳结论，随后强制终止全部成员。

多个任务之间通过并发调度机制实现吞吐效率的最大化。外层编排器以信号量控制同时活跃的团队数量，每个任务的内部协作对编排器而言是一个不透明的执行单元——它只关心启动与收尾，中间的成员交互与迭代修复完全由主控节点自治管辖。任务终态确定后，系统触发一次模型辅助的自动评估，从各成员的行为轨迹和业务产出中提炼多维度的效能分析，为后续的实验比较沉淀可量化的参照素材。

**架构基底**

整套协作修复流程的运转依托三个层面的基础设施：成员间的通信网络、单个智能体的行为驱动框架、以及模型推理与脚本执行的接入形态。三者各自解决一类根本性的工程问题，彼此正交但协同承载了第三章所描绘的团队化工作模式。

**通信网络**

智能体之间的一切信息往来均通过异步消息传递完成，不存在共享内存或直接的函数调用关系。每位成员在加入团队时获得一条专属的收件通道，其他成员通过全局的通道注册表按名称检索目标通道并投递消息。投递操作即时返回不阻塞发送方，而接收操作在通道为空时自动挂起当前协程，直至新消息到达后唤醒——这一机制天然实现了"有事则醒、无事则眠"的空闲调度，无需轮询或定时器介入。

消息本身是强类型的结构化实体，每条携带语义类别、发收双方标识、自由格式的正文以及可选的机器可读载荷。类别标签决定了接收方的处理路径——任务分派、成果交付、修改请求、状态问询等不同意图各有对应的类型编码，使得通信协议在保持灵活性的同时具备可预期的行为语义。

**智能体运行时**

每位成员以独立的异步协程形态存在，共享同一个事件循环但拥有各自隔离的执行上下文。单个智能体的行为节律遵循固定的循环骨架：

- 检查收件通道，将新到达的消息注入对话上下文
- 估算当前上下文的token占用，必要时触发推理链的归档与浓缩
- 携带完整的对话序列和可用工具声明向DeepSeek API发起请求
- 解析模型响应中的工具调用指令，逐一分派至对应的处理方法并收集执行结果
- 将结果拼接回对话序列，持久化当前上下文快照至磁盘

这一骨架在所有角色间通用，差异化的行为通过两个维度注入：角色专属的身份提示词界定了成员的职责边界与协作规约，而技能加载机制则在运行时动态扩充可用工具集——成员启动后仅持有通信、文件读取等基础能力，加载特定技能后方获得与其职责匹配的专业工具，同时技能文档的正文注入上下文成为领域操作的方法论指引。

上下文管理贯穿整个生命周期，采用分层递进的策略控制信息膨胀。覆盖式文件写入完成后即时擦除调用参数中的冗余文本；交互式调试会话关闭时批量移除逐条的命令响应记录，代之以精炼的发现摘要；当token占用逼近阈值时，系统将累积的推理链转储至磁盘并发起独立的模型调用，生成一份第一人称的回顾性概述替换原始的思维痕迹。成员亦可主动触发这一压缩流程以回收上下文空间。所有被精简的原始内容均保留磁盘副本，确保事后审计与评估环节不因运行时的空间优化而丧失信息来源。

**推理与执行的接入**

模型推理能力经由DeepSeek API引入，该平台的两项特性对智能体框架尤为关键。工具调用协议允许模型在对话过程中以强类型的函数调用形式表达操作意图，省去了传统方案中从自由文本解析结构化指令的脆弱环节。思考模式的叠加则令模型在选定工具前先输出一段显式的推理过程，为决策路径提供可观测、可归档的透明记录。并发策略方面，平台未对单用户施加固定的请求频率约束，仅在服务端承压时通过状态码示意暂缓，配合指数退避的重试机制即可充分利用可用带宽。

脚本执行采用子进程隔离模式，每次运行在独立的操作系统进程中启动Python解释器。地理空间处理所需的专用库群部署于与项目主环境分离的虚拟环境中，解释器的选择依据任务元数据中的技术栈属性动态切换——开源任务调度至预配置的独立环境执行，依赖闭源接口的任务则交由ArcGIS Pro自带的解释器承接。执行产物按调用序列归档至层级化的输出目录，涵盖脚本快照、标准输出流、错误流以及追踪钩子生成的诊断文件，为后续的故障分析保全完整的现场证据。
