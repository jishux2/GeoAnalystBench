**地理分析任务导向的多智能体修复：基准整合、协作组织与实验承托**

**项目背景与研究动机**

大语言模型在程序合成领域的能力边界正被持续探测，而地理空间分析作为一类高度领域化的编程场景，对这些边界施加了独特的张力。一段合格的空间处理脚本不仅要求语法层面的正确，还须在坐标参照系的对齐、矢量与栅格数据的异构操作、空间谓词的恰当选取等专业维度上同时达标——任何一环的疏漏都可能导致结果在地理意义上失效，而这种失效往往不会以运行时异常的形式显现，令事后诊断更加棘手。

围绕这一考察方向，学术社区已积淀了若干评测资源。GeoAnalystBench汇集了经GIS领域专家审定的地理处理问题，涵盖城市热岛效应建模、野生动物迁徙追踪、地下水脆弱性评估等多元场景；GeoBenchX以更大的样本规模延伸了覆盖面；GeoColabs、GeoJsonAgent、ShapefileGPT等系列则各自聚焦于特定的数据格式或分析范式。本项目将上述来源统一纳入一套横跨337个问题实例的综合评测体系，借助异构任务族群的交叉覆盖来消解单一数据源固有的采样偏倚。

然而，评测体系所能揭示的仅仅是静态的生成质量切面。在真实的工程实践中，一份首次提交即无瑕的脚本是罕见的例外而非可依赖的常态——更普遍的情形是：初始产出携带着程度各异的缺陷，需要经历多轮定位、修补与验证才能收敛至可用状态。这一观察将关注焦点从"模型能否一次性写对"的孤立命题，推移至一个更贴合实际的问题域：当生成的程序存在瑕疵时，能否通过结构化的协作流程将其逐步修复？

本项目在上述评测语料库之上，构建了一套基于多角色协作的自动化修复系统来回应这一问题。每个地理处理任务被交付给一个由专职成员组成的智能体小组：数据资源的结构性摸底、脚本的全链路实现、运行时故障的系统性诊断与自主修复——各环节由独立的角色承担，在严格界定的权责边界内运作，通过异步消息传递协调彼此的工作节奏。这一设计将评测重心从代码生成能力的单点度量，转向了专业化团队在开放性调试场景中的分工效率与问题解决能力。

围绕这套协作机制，项目同步建立了完整的实验支撑设施。参考实现的批量执行为每个问题实例生成基线产物，智能体团队的运行产出则提供待评判的对照样本，两组材料最终汇入一条模型驱动的评估管线，沿多个质量维度输出结构化的判定结果。从任务语料的索引管理、执行环境的隔离供给，到诊断产物的归档与评分的持久化，各外围模块以统一的批处理框架为骨干串联运转，使得从单任务的深度调试到全集规模的批量实验，都能在同一套基础设施上无缝切换。

**基准测试集的构成**

评测语料以`datasets.json`为载体，采用双层JSON结构组织：外层的元数据区段汇总全集的容量指标，内层的任务数组则逐条承载每个地理处理题目的完整档案。单条档案的信息骨架如下：

| 字段 | 语义角色 |
|------|---------|
| task_ID | 题目在其所属族群内的唯一定位符 |
| task_text | 凝练至一段自然语言的分析诉求陈述 |
| solvable | 可解性的定性判定，沿用早期筛选阶段的遗留标记 |
| used_dataset | 题目运行所仰赖的数据文件索引，形态因题而异 |
| reference_solutions | 预处理后的基线实现代码，缺省时以空序列占位 |
| source | 标识该题目所归属的来源集合 |

九个来源集合在样本规模、标识符编制规则与学科切面上各具辨识度：

| 来源 | 标识符范式 | 容量 | 侧写 |
|------|-----------|-----:|------|
| GeoBenchX | 时间戳复合键，如`TASK_250309_135125_734213` | 200 | 体量居首的综合性语料池 |
| ShapefileGPT | 前缀递增序列，如`sfgpt_1` | 42 | 矢量格式操作的密集演练 |
| GeoJsonAgent | 简明递增序列，如`task_41` | 26 | 以GeoJSON为轴心的处理链路 |
| GeoAnalystBench | 三位数字编号，如`001` | 25 | 领域专家把关的精选题库 |
| GeoColabs | 类别字母缀编号，如`E-8` | 21 | 协同分析情境下的复合任务 |
| Location | 论文溯源键，如`TASK_PAPER_16950049_001` | 10 | 学术文献衍生的定位推理 |
| Mining | 领域前缀序列，如`TASK_MINING_001` | 5 | 采矿工程的垂直应用 |
| Indoor | 时间戳缀编号，如`TASK_260329_165315_001` | 5 | 建筑内部空间的分析场景 |
| PoiMSN | 实体编码缀编号，如`TASK_28052177_001` | 3 | 兴趣点与空间网络的交叉建模 |

同一集合内部的标识符遵循统一的铸造规则，集合之间的命名风貌则泾渭分明。索引模块在加载阶段依据`source`字段将散布的条目重新归拢至各自的族群序列中，修正原始文件中偶现的排列越位。

`used_dataset`的内容组织并非整齐划一。多数题目以扁平序列径直枚举所需的文件名称；另有一部分采用嵌套序列，每层子序列勾勒一种可行的数据配方——同一分析诉求得以经由不同的输入组合抵达终点。预处理阶段对嵌套层执行了去重压缩，剔除内容全同的冗余配方但保留层级骨架，为后续的数据——代码关联留出弹性。极少数条目在此字段下开列空序列，表征其分析路径完全脱离本地文件，转而依托程序化的远端资源获取。

参考实现经历了一轮面向可移植性的统一改造方能入库。原始稿本中，文件寻址的手法纷繁芜杂：有的将路径写死为字符串常量，有的借助`os.path.join`逐级拼装，有的甚至通过`os.chdir`在脚本启动时偷换工作目录的锚点。改造策略以一对环境变量——`TASK_INPUT_ROOT`与`TASK_OUTPUT_ROOT`——取代全部硬编码的位置假设：前者在运行时指向题目所属来源的数据集目录，后者锚定交付产物的汇集口径。经此处理，任何一份参考脚本都不再对自身的物理部署方位持有预设，适配不同安置布局时仅需调节变量注入值而无需触碰代码本体。对于原始语料中附带多份候选解答的题目，仅首个实现被纳入基线序列；未配备任何实现的题目则以空数组如实标注，在下游的评估环节中相应地豁免参考侧的材料供给。

`solvable`字段铭刻着数据集自身的谱系演变。在早期的探索性实验中，模型被赋予自主研判数据充分性的权限——若既有资源不足以支撑指定分析，任务即被裁定为不可完成并标记`REJECT`。随着研究议程从开放式探索转向脚本工程能力的定向考核，拒绝语义的实用价值逐渐消退，绝大多数记录已被重新归入`ACCEPT`阵营。残存的`POSSIBLE`标记折射出一片认知灰区：这些题目的可解前景在当时的判定框架下未能获得确切的裁断，其参考实现的配备状况也因而呈现不规则的缺补分布。

**评测范式**

系统的运转围绕两条并行的执行管线展开。第一条管线驱动参考实现的批量验证：对每个配备了基线代码的题目，在隔离的子进程中注入`TASK_INPUT_ROOT`与`TASK_OUTPUT_ROOT`环境变量后启动脚本，捕获退出状态与输出流，将产物归档至参考基线目录。这一过程不涉及任何模型推理，其价值在于为下游的对比评估储备经过实际运行检验的参照产物。第二条管线将每个题目托付给一组按职能分化的智能体编队，经历从数据勘察到代码落地再到反复诊治的完整工作弧线。两条管线共享同一套调度骨架：任务池的多条件筛选、信号量管控的并发度、逐条目的终态持久化与断点续跑能力，均由通用的批处理框架统一供给。

协作管线内部的运转节律遵循严格的先后次序。探查员率先进入活跃状态，依据任务描述中列明的数据文件清单与来源归属，在数据集仓库的对应目录下逐份执行系统化的属性审查——字段清单、几何特征、坐标参照体系、值域分布——并将全部发现汇编为一份落盘报告。报告的起首处铭记着数据集目录的完整访问路径，为下游消费者免除自行推导之劳。工程师在此期间保持休眠，直至报告抵达后方着手研读。面对报告中未能覆盖的数据疑点，工程师可向探查员发起定向的补充问询，在逐步填补认知缺口的过程中推进流水线的架构规划。当对数据版图的把握足以支撑每一处实现决策时，工程师完成脚本编写并将全部交付产物指派至任务目录下的`pred_results/`子目录，在关键的流水线衔接处植入断言检查点，随后将源文件位置与实现概要一并递交诊断专员，自身即退出活跃状态。脚本中对数据文件的引用采用探查员报告中给出的绝对路径，规避了因目录层级推算失误而引发的寻址偏差。

诊断环节构成修复过程的迭代内核。专员接手脚本后首先发起一次带追踪钩子的运行，借由异常捕获记录和函数调用日志建立初始的故障画像。随后依据诊断产物中暴露的线索形成假设——对于体量较大的输出流与追踪文件，专员优先截取尾段或以模式检索定位关键区域，而非将全文一次性灌入上下文。交互式调试器在崩溃现场或特定代码区域勘察运行时状态，逐步收窄故障范围。确认根因后，专员直接对脚本施加定向修补并重新执行以验证效果；若需在不改动源文件的前提下观测中间变量，则生成一份嵌入诊断语句的临时副本加以运行。对于脱离主脚本执行流程的轻量级机制验证，专员可将代码片段直接传入执行工具获取即时反馈。数据层面的疑点则路由至探查员进行针对性复核。这一"假设——验证——修正"的循环持续推进，直至脚本通过全部运行时校验且业务产出符合任务预期。由于`pred_results/`目录在每次脚本运行时被清空重建，修复告竣时该目录下留存的即为最终版本的交付产物，无需额外的聚敛步骤。

主控节点以确定性的程序逻辑驱动，不介入模型推理。启动时它向三位成员的收件通道预置情境通报——探查员即刻获得完整的任务描述与数据集目录的绝对路径定位，工程师与诊断专员各自收到标注了入场时机的待命通知。此后节点以短周期轮询收件箱，感知团组的运转态势但不主动发起进展问询以避免干扰成员的工作节奏。工程师的首次交付触发初始脚本的基线备份，诊断专员的终态消息驱动结果收集与差异比对——最终产出通过比对基线脚本与当前版本自动生成变更记录，连同诊断专员递交的业务评估和根因分析一并封存。若全局时间预算先行耗尽，节点向诊断专员发送终止指令——此时若专员恰好完成了一轮修补但尚未验证其效果，系统给予额外一次执行的宽裕窗口，使专员得以在提交受迫结论前确认最近一次修正的实际表现。

两条管线的终态确定后，系统可触发一次模型驱动的自动评估。评估模块从参考基线侧汇集脚本原文、执行状态与产物描述，从智能体工作空间侧收集对应的生成代码、最终执行记录与`pred_results/`下的交付文件，将两组材料分别封装为独立的上下文载体后，连同任务描述一并递交评估模型。产物在进入评估视野前经过按类型分流的预处理：矢量、栅格与表格数据由诊断脚本提炼为结构摘要，图像文件转化为可供多模态模型直接感知的编码形式，一般文本按容量阈值截取，未能归入已知类别的文件仅以名称列示。评估模型沿任务契合度、执行有效性、产出完备性、结果可信度、可视化表现力、代码工艺水准六个递进维度输出带有评分与论据的逐项判定，辅以总体评分、改进建议与双方实现的对比分析，持久化至任务目录为后续的跨题目统计聚合沉淀可量化的参照素材。

**架构基底**

整套协作修复流程的运转依托三个层面的基础设施：成员间的通信网络、单个智能体的行为驱动框架、以及模型推理与脚本执行的接入形态。三者各自回应一类根本性的工程诉求，彼此正交但协同承载了前述的团队化工作模式。

**通信网络**

智能体之间的一切信息往来均经由异步队列完成，不存在共享内存或直接的函数调用关系。每位成员在登记就位时获得一条专属的收件通道，其他成员通过全局的通道注册表按名称检索目标通道并投递消息。投递操作即时返回不阻塞发送方，而接收操作在通道为空时自动挂起当前协程，直至新消息到达后唤醒——这一机制天然实现了"有事则醒、无事则眠"的空闲调度，无需轮询或定时器介入。

消息本身是携带语义类别标签的自描述实体，每条附有发收双方标识、自由格式的正文以及可选的机器可读载荷。类别标签决定了接收方的处理路径——任务分派、成果交付、补查请求、终态宣告等不同意图各有对应的类型编码，使得通信协议在保持灵活性的同时具备可预期的行为语义。

**智能体运行时**

每位成员以独立的异步协程形态存在，共享同一个事件循环但拥有各自隔离的执行上下文。单个智能体的行为节律遵循固定的循环骨架：检查收件通道并将新到达的消息注入对话上下文，评估当前上下文的容量占用，携带对话序列和可用工具声明向推理模型发起请求，解析响应中的工具调用指令并逐一分派执行，最后将结果拼接回对话序列并持久化快照至磁盘。

角色间的行为差异通过两个维度注入：身份提示词界定成员的职责轮廓与协作规约，技能加载机制在运行时动态扩充可用工具集。成员启动后仅持有通信、文件读取、内容检索等基础能力，加载特定技能后方获得与其职责匹配的领域工具，同时技能文档的正文汇入上下文成为方法论指引。

文件访问采用按需获取的模式。读取工具支持全量载入、按行偏移的分片提取以及从末尾回溯的尾段截取三种姿态，辅以文件体积门槛与结果字符量的双层守卫防止过大的内容一次性涌入上下文。内容检索工具基于ripgrep提供模式匹配与文件枚举两种能力，与分片读取配合形成"定位——截取——审阅"的高效检索链路。位于任务目录之外的数据集仓库被排除在直接读取范围之外，须经由探查员的诊断脚本间接获取所需信息。

上下文管理贯穿整个生命周期。工具结果预算为每次返回设定体积上限，超出阈值的内容自动转存磁盘并以截断预览替代。调试会话关闭时，逐条的命令响应记录被批量移除，代之以凝练的发现摘要与磁盘归档路径。当容量占用逼近预设阈值时，系统将推理链提取转储并经由独立的模型调用浓缩为回顾性概述，替换原始的冗长思维痕迹（若此举仍显吃紧，早期积存的大体积读取与检索返回亦被精简为附带重取指引的简短凭据，新一轮推理链的积累由此重获空间"）。所有被裁汰的原始内容均保留磁盘副本，确保事后审计不因运行时的空间优化而丧失信息来源。

**推理与执行的接入**

模型推理能力经由两条通路引入。智能体团队的工作循环对接DeepSeek API，该平台的工具调用协议允许模型以函数调用形式表达操作意图，思考模式的叠加则令模型在选定工具前先输出显式的推理过程，为决策路径提供可观测的透明记录。评估管线则接入一个本地部署的反向代理端点，背后承载的模型具备多模态感知能力，能够在评判环节直接处理图像形态的产物而无需将其转译为文本描述。两条通路在协议层面均对齐OpenAI Chat Completions规范，差异仅体现在端点地址、模型标识与传输模式上。

脚本执行采用子进程隔离模式，每次运行在独立的操作系统进程中启动Python解释器。地理空间处理所需的专用库群部署于与项目主环境分离的虚拟环境中。参考代码执行时，框架通过环境变量向子进程注入数据源目录与产物输出目录的绝对路径，并强制设定非交互式的图形渲染后端以杜绝弹窗阻塞；智能体生成的脚本则依据探查员报告中记载的路径信息自行完成数据寻址，产物写入工程师在编码阶段约定的固定子目录。执行产物按调用序列归档至层级化的输出目录，涵盖脚本快照、标准输出流、错误流以及追踪钩子生成的诊断文件，为后续的故障分析保全完整的现场证据。
