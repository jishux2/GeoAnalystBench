**DeepSeek API技术参考与服务端对接**

**API服务概况**

DeepSeek平台维护两套服务端点，分别对应常规功能与实验性特性：

| 端点类别 | 地址 | 适用范围 |
|---------|------|---------|
| 标准端点 | `https://api.deepseek.com` | 对话补全、工具调用、思考模式 |
| Beta端点 | `https://api.deepseek.com/beta` | 试验性功能（含前缀续写等） |

两套入口在响应格式与错误处理方面保持一致，区别仅体现在所接纳的请求参数集合上。

模型层面提供两个标识符，对应不同的默认行为配置：

- `deepseek-chat`——DeepSeek-V3.2的常规对话模式，不主动展开推理链，适合需要直接产出结果的场景。默认输出上限为4K token，可经`max_tokens`调至8K。
- `deepseek-reasoner`——同一底座的思考模式变体，在生成最终回答前先输出一段显式的推理过程。默认输出容量扩展至32K，上限可达64K。

值得注意的是，思考模式并非`deepseek-reasoner`的专属能力。通过在请求体中配置`thinking`参数，`deepseek-chat`同样能够激活推理链输出，此时模型行为与`deepseek-reasoner`趋于一致。这意味着开发者可以在单一模型标识符下灵活切换两种工作状态，无需为不同场景分别维护调用逻辑。

接口规范层面，DeepSeek全面对齐OpenAI Chat Completions协议——从请求体的消息序列与工具声明编排，到响应体的候选结果嵌套结构，均可与面向OpenAI设计的客户端代码无缝互换，仅需调整服务端点地址。上下文窗口容量为128K token，这一额度涵盖输入消息序列与输出生成的合计消耗，构成了智能体上下文管理策略——推理链压缩的触发阈值、工具结果的体积预算——的底层物理约束。

计费方面采用按token计量的阶梯模式：输入侧依据缓存命中情况区分定价，命中时0.2元/百万token，未命中则为2元/百万token；输出侧统一按3元/百万token结算。并发层面未对单用户施加固定的请求频率约束，仅在服务端承压时通过503状态码示意调用方适度退让，这一特性为批量推理场景下的吞吐效率提供了充裕的弹性空间。

**请求与响应规范**

所有调用模式共享一组基础参数，构成请求体的骨干结构：

`messages`承载完整的对话序列，每个元素包含`role`与`content`两个必备属性。角色取值涵盖`system`、`user`、`assistant`及工具调用场景下的`tool`四种类型，按时序交替排列以还原多轮交互的完整脉络。

`model`指定目标模型标识符，其取值决定了默认的输出行为与容量配额。

`temperature`调节采样过程的随机程度，取值区间为0到2。数值偏低时输出趋于确定性，偏高则引入更多变异。需要留意的是，一旦启用思考模式，该参数将被静默忽略——即便显式设定也不会对生成结果产生实际影响。

`max_tokens`划定单次响应的长度天花板。在思考模式下，这一预算涵盖推理链与最终回答的合计消耗，需据此预留充足的空间。

`stop`定义终止序列，模型在输出匹配该序列的文本时立即截断生成。这一机制在约束输出格式时尤为有效——例如将停止序列设为代码块的闭合标记，即可确保模型仅产出纯粹的程序文本而不附带额外阐释。

`stream`以布尔值控制传输模式，设为`false`时服务端在生成全部完成后一次性返回完整结果。

响应体沿袭前述的协议兼容框架，核心内容嵌套于`choices[0].message`路径下。常规模式中`content`字段承载生成文本；思考模式下额外出现与之同级的`reasoning_content`字段，记录模型的中间推理过程；工具调用场景则通过`tool_calls`数组传递结构化的函数调用指令。

**前缀续写机制**

Beta端点开放了一项输出格式约束能力：在请求的消息序列末尾附加一条标记`prefix: true`的assistant消息，其内容将被视为模型回复的既定开头，后续生成从该锚点接续展开。搭配终止序列的配合使用，可将模型的产出范围精确框定在预设的起止边界之内。启用时须遵守两项约束：前缀消息必须位于数组的末端位置，且请求须投递至Beta端点。当前系统的代码编写由脚本工程师在智能体的工具调用循环中自主完成，未采用此机制。

**工具调用与思考模式**

工具调用机制赋予模型在对话过程中主动触发外部函数的能力，是构建智能体交互循环的协议基础。DeepSeek的实现遵循OpenAI兼容规范，通过请求体中的两个专属参数来配置这一行为。

`tools`参数接收一组函数声明，每个条目描述一项模型可支配的操作资源。声明结构包含三个层面的信息：

- `name`标识调用入口
- `description`阐明该函数的职能与适用场景，供模型在决策时进行语义匹配
- `parameters`以JSON Schema格式约定输入参数的类型与约束

模型依据对话上下文与函数描述的关联程度，自行判断是否发起调用以及选取哪个函数。单次请求至多声明128个函数。

`tool_choice`参数则从调用方视角施加行为管控：

| 取值 | 效果 |
|------|------|
| `none` | 完全屏蔽工具触发，模型仅产出文本回复 |
| `auto` | 授权模型自主抉择，既可生成消息也可发起调用 |
| `required` | 强制模型必须激活至少一项工具 |
| `{"type": "function", "function": {"name": "..."}}` | 锁定特定函数作为调用目标 |

未显式设定时，有工具声明则默认为`auto`，无声明则回落至`none`。

**多轮交互的消息流转**

工具调用场景下的对话序列遵循固定的递进规程：

1. 模型决定调用函数时，响应中的`tool_calls`数组承载具体指令，每个条目包含函数名、序列化的参数载荷以及系统分配的调用标识
2. 调用方执行实际的函数逻辑后，将结果以`role`为`tool`的消息形式回填至对话序列，`tool_call_id`与先前的调用标识配对，`content`携带执行产出
3. 扩展后的消息序列再次递交模型以驱动下一步决策

如此往复，直至模型不再发起工具调用而是给出最终的文本回答。

**思考模式的叠加启用**

思考模式可与工具调用并行运作，为模型在决策环节引入显式的推理过程。激活方式有两条路径：

- 将`model`参数直接设为`deepseek-reasoner`
- 在使用`deepseek-chat`时附加`thinking`参数配置为`{"type": "enabled"}`

后者的灵活性更高，允许同一模型标识符在常规对话与推理增强之间按需切换。

启用后，响应结构中在`content`与`tool_calls`之外额外呈现`reasoning_content`字段，记录模型的中间思考轨迹。几项配套约束值得留意：

- `temperature`等采样参数不再生效——即便显式传入也会被静默忽略
- `max_tokens`的预算涵盖推理链与最终输出的合计消耗，配置时须为两部分留足余量
- 历史消息中累积的`reasoning_content`会持续占用传输带宽与上下文窗口，官方推荐在构造后续请求前将已处理轮次的推理内容予以剥离

最后一点在调试智能体的长程交互场景中尤为关键——每轮工具调用都会扩展消息序列，若不及时清理推理文本，上下文膨胀的速度将远超有效信息的积累。

**严格模式**

Beta端点还引入了Strict Mode作为工具调用的格式保障层。在函数声明中将`strict`属性设为`true`后，API会强制校验模型输出的参数载荷，确保其与预定义的JSON Schema达成零偏差的结构对齐。

该模式支持的类型定义涵盖`object`、`string`、`number`、`integer`、`boolean`、`array`、`enum`、`anyOf`，以及通过`$ref`与`$def`实现的模式复用与递归结构。启用时须遵守一项附加约束：所有`object`类型的属性必须声明为`required`，且须将`additionalProperties`设为`false`。

这一限制虽然收紧了Schema的编写自由度，却从根本上消除了模型输出结构偏移的风险。对于依赖精确参数格式的自动化流程而言，是值得采纳的可靠性增强手段。

**错误码体系与重试策略**

API调用过程中遭遇的异常通过HTTP状态码传递分类信号，依据其可恢复性划分为两个阵营。

客户端错误源于请求自身的配置缺陷，重试无法改善状况，应直接终止并上报：

| 状态码 | 语义 | 典型成因 |
|--------|------|---------|
| 400 | 请求体格式畸形 | JSON结构损坏或必填字段缺失 |
| 401 | 身份凭证无效 | API密钥错误、过期或未携带 |
| 402 | 账户额度耗尽 | 余额不足以支撑本次调用 |
| 422 | 参数约束违规 | 字段取值超出许可范围或类型不匹配 |

服务端错误反映的是平台侧的临时性状况，经过适当等待后通常可自行恢复：

| 状态码 | 语义 | 应对姿态 |
|--------|------|---------|
| 429 | 请求频率过密 | 尽管平台未设固定限额，突发流量仍可能触发保护机制 |
| 500 | 内部处理故障 | 服务端运行时异常 |
| 503 | 负载容量饱和 | 系统资源暂时紧张，稍后重试即可 |

针对后一类状况，指数退避策略在实践中表现稳健。其核心思路是让连续失败之间的等待间隔逐次翻倍——首次受挫后暂停2秒，第二次延至4秒，第三次拉长到8秒——通过递增的冷却周期缓解对服务端的集中冲击。若三轮尝试后仍未获得有效响应，则将该次调用标记为失败并记录诊断信息，交由上层逻辑决定后续处置。

除HTTP层面的错误外，网络中断与请求超时也需纳入容错范畴。前者表现为连接建立阶段的通信异常，后者则是服务端在规定时限内未能完成生成。两种情形均适用相同的退避重试逻辑，共享最大尝试次数的全局预算。
